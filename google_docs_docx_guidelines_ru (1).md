# Шаблон требований к DOCX-документам для Google Docs

Версия: 1.0  
Назначение: короткая инструкция для будущих сессий, чтобы получать DOCX-файлы, которые нормально открываются, редактируются и читаются в Google Docs.

---

## 1. Короткий промпт для будущих задач

Скопируй этот блок в начало запроса, когда нужен DOCX:

```text
Создай редактируемый DOCX-документ, оптимизированный для Google Docs.

Требования к оформлению:
- Основной текст: Arial или Calibri, 11 pt, line spacing 1.15, интервал после абзаца 6 pt.
- Не использовать основной текст 8 pt. В таблицах минимум 10 pt, в источниках минимум 9.5–10 pt.
- Использовать встроенные стили: Title, Heading 1, Heading 2, Heading 3, Normal, Table.
- Документ должен быть удобен для редактирования в Google Docs: без floating text boxes, сложных фигур, фоновых картинок, нестандартных шрифтов и ручного выравнивания пробелами.
- Таблицы должны быть читабельными: не более 4–6 колонок на обычной странице; широкие таблицы разбивать на несколько таблиц или выносить в appendix.
- Все ссылки делать кликабельными; длинные URL не вставлять в основной текст, а выносить в Sources Appendix.
- Добавить титульную страницу, дату, версию, executive summary, оглавление-заглушку, основные разделы, выводы и appendix с источниками.
- Перед выдачей визуально проверить рендер DOCX: страницы не должны иметь обрезанный текст, наложения, сломанные таблицы или нечитаемые шрифты.

Выдай итоговый файл в формате .docx.
```

---

## 2. Базовая структура документа

Рекомендуемый порядок:

1. **Title page**
   - Название документа.
   - Подзаголовок / объект исследования.
   - Дата исследования.
   - Версия документа.
   - Статус: Draft / Final / Internal.

2. **Document control**
   - Автор / подготовлено для.
   - Дата обновления.
   - Ограничения исследования.
   - Confidence level, если документ аналитический.

3. **Table of Contents placeholder**
   - В DOCX можно оставить строку: `Оглавление вставить в Google Docs: Insert → Table of contents`.
   - Не нужно вручную делать длинное оглавление, если документ дальше редактируется.

4. **Executive Summary**
   - 5–10 коротких тезисов.
   - Главный вывод вверху.
   - Для due diligence: Go / Conditional Go / No-Go.

5. **Main body**
   - Разделы через Heading 1.
   - Подразделы через Heading 2.
   - Детали через Heading 3.
   - Таблицы и списки вместо длинных абзацев.

6. **Decision / Recommendation**
   - Четкая рекомендация.
   - Условия, ограничения, следующие шаги.

7. **Appendix**
   - Sources Appendix.
   - Risk register.
   - Methodology.
   - Additional tables.

## 3. Рекомендуемые стили

| Элемент | Стиль | Размер | Формат |
|---|---:|---:|---|
| Title | Title | 22–26 pt | Bold, темный цвет, без перегруза |
| Subtitle | Subtitle | 13–15 pt | Regular / semi-bold |
| Heading 1 | Heading 1 | 16–18 pt | Bold, page break before optional |
| Heading 2 | Heading 2 | 13–14 pt | Bold |
| Heading 3 | Heading 3 | 11.5–12 pt | Bold |
| Body text | Normal | 11 pt | Line spacing 1.15, after 6 pt |
| Bullet list | Normal | 11 pt | Короткие пункты, без вложенности глубже 2 уровней |
| Table header | Table style | 10.5–11 pt | Bold, светлая заливка |
| Table body | Table style | 10–11 pt | Не ниже 10 pt |
| Caption / note | Caption | 9.5–10 pt | Только для коротких примечаний |
| Sources appendix | Normal / small | 9.5–10 pt | Кликабельные ссылки |

**Главное правило:** 11 pt для основного текста. 8 pt не использовать, кроме редких технических сносок, но лучше даже там держать 9.5–10 pt.

---

## 4. Шрифты

Лучшие варианты для Google Docs:

- **Arial** — самый безопасный вариант.
- **Calibri** — хорошо переносится из Word, но в Google Docs может выглядеть немного иначе.
- **Aptos** — современный Word-шрифт, но совместимость зависит от среды.
- **Roboto** — хорошо смотрится в Google Docs, но лучше использовать только если точно доступен.

Не использовать:

- Кастомные корпоративные шрифты без необходимости.
- Встроенные font files.
- Слишком декоративные шрифты.
- Смешение 4–5 разных шрифтов в одном документе.

Рекомендация по умолчанию: **Arial 11 pt для body**.

---

## 5. Макет страницы

Рекомендуемые настройки:

| Параметр | Значение |
|---|---|
| Page size | A4, если не указано иное |
| Orientation | Portrait |
| Margins | 18–22 mm со всех сторон |
| Header / footer | Минимальные: название документа, дата, номер страницы |
| Page breaks | Перед крупными разделами |
| Line spacing | 1.15 |
| Paragraph spacing | 6 pt after, 0 pt before для body |

Не делать:

- Сложные фоны на страницах.
- Много колонок как в журнале.
- Текстовые блоки поверх картинок.
- Ручное выравнивание пробелами.
- Слишком узкие поля ради экономии страниц.

---

## 6. Таблицы

Правила для хорошей читаемости:

- Максимум 4–6 колонок на portrait-странице.
- Если колонок больше — разбить таблицу на несколько тематических таблиц.
- Заголовок таблицы должен повторяться при переносе на новую страницу.
- Шрифт в таблицах: **10–11 pt**, не 8 pt.
- Внутренние отступы ячеек должны быть достаточными, чтобы текст не прилипал к границам.
- Длинные ссылки не держать в таблицах; использовать короткий label и ссылку.
- Не злоупотреблять merged cells: они часто ломают редактирование в Google Docs.
- Для risk matrix использовать короткие значения: Low / Medium / High / Critical.

Хорошие таблицы:

- Короткие строки.
- Понятные заголовки.
- Одна мысль на строку.
- Не более 1–2 цветов заливки.
- Без чрезмерно длинных абзацев внутри ячейки.

---

## 7. Цвета и визуальный стиль

Использовать ограниченную палитру:

| Назначение | Цветовой подход |
|---|---|
| Заголовки | Темно-синий, темно-серый или черный |
| Акценты | Один дополнительный цвет |
| Табличные заголовки | Светло-серая или светло-синяя заливка |
| Low risk | Зеленый / светло-зеленый |
| Medium risk | Желтый / янтарный |
| High risk | Оранжевый / красный |
| Critical risk | Темно-красный |

Правила:

- Цвет должен помогать чтению, а не украшать ради украшения.
- Не использовать кислотные цвета.
- Не полагаться только на цвет: рядом должен быть текстовый label, например `High`.
- Проверять контрастность: светлый текст на светлом фоне плохо читается.

---

## 8. Источники и ссылки

Правильная практика:

- Все факты, которые могут быть спорными или меняющимися, должны иметь ссылку.
- Ссылки должны быть кликабельными.
- В основном тексте лучше писать название источника, а не длинный URL.
- Полные ссылки выносить в **Sources Appendix**.
- Не использовать неподтвержденные данные как факт.
- Разделять: факт, оценка, гипотеза, вывод.

Формат источника:

```text
Source name — short description — URL — access date
```

Пример:

```text
Real Madrid — Roberto Carlos official legend profile — [clickable URL] — accessed 2026-05-12
```

---

## 9. Что обязательно просить у модели

Для аналитических документов просить:

- Дату исследования.
- Методологию.
- Assumptions.
- Ограничения исследования.
- Таблицы с фактами.
- Отдельные выводы.
- Confidence level по ключевым блокам.
- Sources Appendix.
- Risk matrix, если есть риски.
- Executive summary на первой странице после title page.

Для документов с большим количеством данных просить:

- Сначала краткий executive summary.
- Потом подробности.
- Затем appendix.
- Таблицы не перегружать; слишком большие таблицы переносить в appendix.

---

## 10. Чего избегать

Не использовать:

- Body text 8 pt.
- Таблицы шрифтом 8 pt.
- Длинные стены текста без заголовков.
- Слишком много жирного текста.
- Более 2 уровней вложенных списков.
- Много разных цветов.
- Floating text boxes.
- Фигуры с текстом вместо обычных абзацев и таблиц.
- Сложные watermarks и фоновые изображения.
- Нестандартные шрифты.
- Ручные переносы строк внутри каждого предложения.
- Ссылки длиной на полстраницы.
- Непроверенные источники без маркировки.
- Дизайн, который красиво выглядит в PDF, но плохо редактируется в Google Docs.

---

## 11. QA-чеклист перед выдачей DOCX

Перед финальной отправкой проверить:

- [ ] DOCX открывается без ошибок.
- [ ] Все страницы визуально отрендерены и просмотрены.
- [ ] Нет обрезанного текста.
- [ ] Нет наложения текста на таблицы / изображения.
- [ ] Все таблицы читаемы.
- [ ] Body text 11 pt.
- [ ] Таблицы минимум 10 pt.
- [ ] Заголовки сделаны через Heading styles.
- [ ] Ссылки кликабельны.
- [ ] Источники вынесены в appendix.
- [ ] Нет случайных внутренних технических токенов или мусора.
- [ ] Нет лишних комментариев / tracked changes, если они не нужны.
- [ ] Метаданные очищены, если документ конфиденциальный.
- [ ] Документ нормально импортируется в Google Docs.

---

## 12. Рекомендуемый формат финальной выдачи

Когда модель отдает файл, хорошо просить такой ответ:

```text
Готово: [скачать DOCX].
Файл оптимизирован для Google Docs: body 11 pt, структурные заголовки, читаемые таблицы, кликабельные источники и appendix.
После загрузки в Google Docs можно вставить автоматическое оглавление через Insert → Table of contents.
```

## 13. Мини-шаблон структуры для due diligence report

```text
A. Executive Summary
B. Recommendation
C. Methodology & Limitations
D. Profile / Background
E. Key Findings
F. Commercial Value
G. Risks & Red Flags
H. Legal / Compliance Review
I. Category-Specific Assessment
J. Competitor Benchmark
K. SWOT
L. Risk Matrix
M. Contractual Safeguards / Next Steps
N. Final Decision
O. Sources Appendix
```

---

## 14. Практическое правило

DOCX для Google Docs должен быть не “красивой картинкой”, а **чистым редактируемым документом**: простая сетка, нормальные стили, 11 pt основной текст, аккуратные таблицы, кликабельные ссылки и минимум сложных объектов.
